# Unit Device ID Namespacing

## Summary

Introduce a unit-aware device identity convention for config-driven simulator UI files so the same local device name can safely appear in multiple units. Future `DeviceUi_*.json` files should allow natural per-unit device IDs such as `Q11`, `B13`, or `QC_B24`, while the application generates globally unique WPF/control IDs internally.

## Problem

The current config-driven UI uses `uiId` directly as a global WPF name prefix. If two unit files both define a device with the same `uiId`, generated controls collide. For example, `DeviceUi_BL170.json` and `DeviceUi_BL180.json` could both define `Q11`, producing duplicate names like `Q11_SimFbButton`.

This is likely with 22 units because industrial equipment names repeat across units.

## Goals

- Allow repeated local device names across units.
- Preserve clean, human-readable JSON for unit authors.
- Generate globally unique UI/control IDs automatically.
- Keep exact PLC tag names unchanged.
- Maintain compatibility with existing `FindName(...)`-based device wiring until the simulator moves fully to MVVM.
- Add validation so collisions fail early with actionable errors.

## Non-Goals

- Do not redesign the full simulator UI in this change.
- Do not replace all hardcoded base/common devices.
- Do not rename PLC tags or normalize PLC symbols.
- Do not remove the existing BL170 Q21 skip rule unless a later spec replaces it with config-level enable/disable behavior.
- Do not introduce a new JSON library unless required by validation/schema work.

## Current Behavior

- Unit files are discovered from `DeviceUi_*.json`.
- The unit ID comes from the filename, for example `DeviceUi_BL170.json` maps to unit `BL170`.
- Each device has `uiId`, and generated controls use either explicit `uiControls` names or fallback names based on `uiId`.
- Device constructors receive `uiId` so existing classes can locate generated controls using `FindName(...)`.
- Duplicate `uiId` values across unit files are not rejected.
- If explicit `uiControls` values are duplicated across files, the app can bind to the wrong control or skip registration.

## Proposed Architecture

Treat device identity as two-layered:

- `unitId`: derived from the config filename, such as `BL170`.
- `deviceId`: local per-unit identity, such as `Q11`.
- `globalUiId`: generated by the loader as `{unitId}_{deviceId}`, such as `BL170_Q11`.

For backward compatibility, keep accepting `uiId` in JSON initially, but interpret it as the local device ID unless it already starts with the unit prefix.

The loader should expose normalized runtime definitions with:

- original `unitId`
- original local device ID
- generated `globalUiId`
- exact PLC tags from config
- generated or validated control names

## FRs

- FR1: Load every `DeviceUi_*.json` file with a filename-derived `unitId`.
- FR2: For every configured device, compute a globally unique `globalUiId`.
- FR3: If JSON `uiId` already starts with `{unitId}_`, keep it unchanged as `globalUiId`.
- FR4: If JSON `uiId` does not start with `{unitId}_`, generate `globalUiId = unitId + "_" + uiId`.
- FR5: Generate fallback `uiControls` names from `globalUiId`, not raw local `uiId`.
- FR6: Pass `globalUiId` into `Analog`, `Digital`, and `Valve` constructors.
- FR7: Keep `plcTag`, `controlTag`, feedback tags, and marker fallback tags exact and unmodified.
- FR8: Detect duplicate `globalUiId` values across all loaded configs and fail startup with a clear message.
- FR9: Detect duplicate generated or explicit control names across all loaded configs and fail startup with a clear message.
- FR10: Continue skipping BL170 `Q21` as currently implemented until a separate config-enabled/disabled rule replaces it.

## NFRs

- Keep implementation surgical and compatible with the current WPF app.
- Do not require network access.
- Errors should be readable to a simulator maintainer, not only a C# developer.
- Runtime behavior should remain deterministic regardless of file system enumeration order.
- Avoid breaking existing `DeviceUi_BL170.json`.

## Acceptance Criteria

- Two different unit configs can use the same local `uiId` without WPF name collisions.
- Existing BL170 config continues to work.
- The simulator project builds successfully.
- Duplicate generated global IDs or control names fail with a clear error.
- Exact PLC tag spelling remains unchanged.
- `Q21` in `DeviceUi_BL170.json` remains ignored unless explicitly changed in a later task.

## Open Questions

- Should duplicate PLC tags across different units be rejected, warned, or allowed for intentionally shared/common devices?
- Should the JSON field eventually be renamed from `uiId` to `deviceId`, or should `uiId` remain for compatibility?
- Should disabled/skipped devices be represented in config with an `enabled` field instead of hardcoded skip rules?
